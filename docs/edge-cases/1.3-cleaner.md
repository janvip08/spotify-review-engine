# Edge Cases — Sub-Phase 1.3: Cleaner (Raw-Level)

> Covers: Deduplication, encoding normalisation, date enforcement, schema validation, and relevance flagging.

---

## Deduplication

### EC-1.3.01 — Exact Duplicate Text Across Sources

| Aspect | Detail |
|--------|--------|
| **Trigger** | A user posts the same review on both Play Store and Reddit, or App Store reviews appear in multiple country feeds |
| **Risk** | Inflated counts; skewed analysis |
| **Mitigation** | Hash key = `sha256(text.strip().lower())`. If the same hash appears with different `source` values, keep the one from the more authoritative source (Play Store > App Store > Reddit). Log the drop. |

### EC-1.3.02 — Near-Duplicate Text (Fuzzy Duplicates)

| Aspect | Detail |
|--------|--------|
| **Trigger** | Slight variations: "This app sucks" vs "This app sucks!" vs "this app sucks" |
| **Risk** | Exact-hash dedup misses these |
| **Mitigation** | Phase 1 approach: normalise before hashing — `text.strip().lower()`, remove trailing punctuation. Phase 2 may add MinHash/LSH-based fuzzy dedup. |

### EC-1.3.03 — Duplicate Reviews Across Runs (Re-Fetch Scenario)

| Aspect | Detail |
|--------|--------|
| **Trigger** | Pipeline is re-run and fetches the same reviews again; appended to existing `reviews_raw.jsonl` |
| **Risk** | Doubled dataset |
| **Mitigation** | 1. Default behavior: **overwrite** `reviews_raw.jsonl` on each run (not append). 2. If append mode is explicitly enabled (future scheduler), load existing hashes from the file first and skip any incoming record whose hash already exists. |

### EC-1.3.04 — Same Text, Different Dates

| Aspect | Detail |
|--------|--------|
| **Trigger** | A user edits their review (Play Store allows edits) — same text hash but different `date` |
| **Risk** | Dedup drops the updated version |
| **Mitigation** | When duplicates are found with the same `(source, text_hash)`, keep the one with the **most recent** date. |

## Date & Time-Range Enforcement

### EC-1.3.05 — Reviews Outside the 4-Month Window Slipping Through

| Aspect | Detail |
|--------|--------|
| **Trigger** | Fetcher's time filter is approximate (e.g., Reddit `t=year` fetches up to a year); extractor doesn't filter |
| **Risk** | Old reviews pollute the dataset |
| **Mitigation** | Cleaner applies a **hard date-range check**: `cutoff = datetime.utcnow() - timedelta(days=120)`. Any record with `date < cutoff` is dropped. This is the **final safety net**. |

### EC-1.3.06 — Reviews With Future Dates

| Aspect | Detail |
|--------|--------|
| **Trigger** | Timezone conversion errors or API bugs produce dates in the future |
| **Risk** | Invalid data; confusing date-range in meta |
| **Mitigation** | Drop any record where `date > datetime.utcnow() + timedelta(hours=24)`. The 24-hour buffer accounts for timezone edge cases. Log these. |

### EC-1.3.07 — Records With `null` Date (Date Missing)

| Aspect | Detail |
|--------|--------|
| **Trigger** | Extractor couldn't parse the date and set it to `null` |
| **Risk** | Cannot enforce time-range; unknown provenance |
| **Mitigation** | **Drop the record.** Date is critical for the 4-month window constraint and for future citation. Log the count of dropped dateless records per source. |

## Encoding & Text Quality

### EC-1.3.08 — Non-UTF-8 Characters / Mojibake

| Aspect | Detail |
|--------|--------|
| **Trigger** | Text contains Latin-1 or Windows-1252 characters that weren't decoded properly |
| **Risk** | Garbled text (`â€™` instead of `'`) |
| **Mitigation** | 1. Attempt `ftfy.fix_text()` (if available) to repair common mojibake patterns. 2. Fallback: encode to UTF-8 with `errors='replace'` to swap undecodable bytes with `�`. 3. If >20 % of characters are replacement chars, drop the record. |

### EC-1.3.09 — BOM Markers and Null Bytes

| Aspect | Detail |
|--------|--------|
| **Trigger** | Raw text starts with `\ufeff` (UTF-8 BOM) or contains `\x00` null bytes |
| **Risk** | JSON serialization issues; display problems |
| **Mitigation** | Strip BOM: `text = text.lstrip('\ufeff')`. Remove null bytes: `text = text.replace('\x00', '')`. |

### EC-1.3.10 — Non-English Reviews

| Aspect | Detail |
|--------|--------|
| **Trigger** | Despite requesting English content, some reviews are in Spanish, Portuguese, Hindi, etc. |
| **Risk** | Non-English text won't be useful for English-language RAG analysis |
| **Mitigation** | **Phase 1 (current):** Do **not** filter by language in the Cleaner — this is a lightweight pre-clean step. Add a `"lang_detected"` field set to `null`. **Phase 2 (deep clean):** Apply `langdetect` or `fasttext` and filter to English-only. Rationale: language detection libraries are heavyweight; keep Phase 1 fast. |

### EC-1.3.11 — Reviews That Are Only Emojis or Special Characters

| Aspect | Detail |
|--------|--------|
| **Trigger** | Text like "👍👍👍", "★★★★★", "...", "---" |
| **Risk** | No meaningful text for analysis |
| **Mitigation** | Check: `len(re.sub(r'[^a-zA-Z]', '', text)) < 3`. If fewer than 3 alphabetic characters, drop the record. |

### EC-1.3.12 — Extremely Short Reviews

| Aspect | Detail |
|--------|--------|
| **Trigger** | Text is "good", "ok", "bad" (1–3 words) |
| **Risk** | Low analytical value, but technically valid |
| **Mitigation** | **Keep in Phase 1.** These are valid data points (especially with a rating). Phase 2 relevance filtering may drop them if they don't relate to discovery/recommendations. |

## Schema Validation

### EC-1.3.13 — Missing `source` Field

| Aspect | Detail |
|--------|--------|
| **Trigger** | Bug in extractor; record has no `source` value |
| **Risk** | Cannot attribute the review; breaks downstream grouping |
| **Mitigation** | **Drop the record.** `source` is a required field. This indicates an extractor bug — log at ERROR level. |

### EC-1.3.14 — `rating` Is Present But Not an Integer

| Aspect | Detail |
|--------|--------|
| **Trigger** | Rating came through as `"4"` (string), `4.5` (float), or `"N/A"` |
| **Risk** | Schema violation |
| **Mitigation** | Attempt `int(float(rating))`. If that fails, set `rating = null`. |

### EC-1.3.15 — `url` Field Contains Malformed URL

| Aspect | Detail |
|--------|--------|
| **Trigger** | URL is a relative path, has missing scheme, or is clearly invalid |
| **Risk** | Broken citations in future RAG chatbot |
| **Mitigation** | Validate with `urllib.parse.urlparse()`. Must have both `scheme` (http/https) and `netloc`. If invalid, set `url = null` rather than dropping the record. |

## Relevance Flagging

### EC-1.3.16 — Source Returns <10 % of Its Cap

| Aspect | Detail |
|--------|--------|
| **Trigger** | After cleaning, Play Store has only 80 records (cap: 1,500) |
| **Risk** | Search terms or source may be misconfigured |
| **Mitigation** | Log a `WARNING`: `"play_store returned only 80/1500 records (5.3%). Consider adjusting search terms or time range."` Include this warning in the per-source `meta.json` under an `"alerts"` array. |

### EC-1.3.17 — Majority of Reviews Are Irrelevant

| Aspect | Detail |
|--------|--------|
| **Trigger** | Quick keyword check shows <20 % of reviews mention discovery/recommendation-related terms |
| **Risk** | Low signal-to-noise ratio |
| **Mitigation** | Run a lightweight keyword scan (not a filter): check for terms like `discover`, `recommend`, `playlist`, `algorithm`, `repeat`, `same songs`, `new music`. Log the relevance ratio in meta. **Do not drop records** — Phase 2 handles relevance filtering. |
